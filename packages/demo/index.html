Focus & keyboard accessibility

Check the tab order in a real browser, not a guess

Focus Pocus works out the exact path the Tab key takes through a page, then grades that path. It checks that the order is right and that every stop is reachable, visible, and announced. It runs in real Chromium only, because that is the one place the answer is true. It builds on tabbable, with a rules layer on top.

Why real-browser only

Any honest focus check needs CSS layout. It has to know what is really visible, what is clipped, and where each control sits on screen. jsdom has no layout engine, so a green jsdom test cannot promise the order holds up in a real browser. Focus Pocus reads live visibility and bounding rects, so a passing check reflects what a keyboard user will actually get.

This is a deliberate choice, not a missing feature. The analyzer stays light on dependencies (only tabbable), so it runs anywhere a real DOM does, including inside a Playwright page.evaluate.

How it decides what's valid

What it checks

A tab order is more than a list. Focus Pocus asks four questions about it, and each rule it ships answers one of them.

  • Order is about whether focus follows the reading order, or hops around because of a positive tabindex or a mismatch between the DOM and the layout.
  • Reachability is about whether the keyboard can get to every interactive control, and whether it avoids the dead stops it should not land on.
  • Visibility is about whether every tab stop is actually painted, or is tabbable while invisible because of opacity, an off-screen position, or clipping.
  • Announcement is about whether every stop has an accessible name, and whether it is free of the aria-hidden and nesting traps that a screen reader chokes on.

A drop-in overlay

The same analysis powers a live overlay you can drop on any page. It numbers the real tab path and draws the hops between stops, turning a hop red when it runs against reading order. It rings every finding in place, and each ring has a tooltip that names the rule, the accessible name, and the role. 

import { reveal } from "@focuspocus/reveal";

// No root, so it analyzes the whole document, exactly like tabbing the page.
const overlay = reveal();

That is the whole integration. The overlay re-analyzes itself on every DOM mutation, so it keeps up with menus opening, modals mounting, and even a virtual list that rewrites its rows every frame. The two demo pages are nothing but this call.

The packages

A small monorepo. The analyzer is pure and framework-free. Everything visual is a separate layer on top of it.

Package What it is
@focuspocus/core The analyzer. It wraps tabbable, applies the rules, and returns a plain result. No framework or test-runner deps. API →
@focuspocus/reveal The framework-agnostic live overlay, built on the analyzer. API →
@focuspocus/demo This site. The docs plus the two interactive demo pages.

A test-runner matcher (Vitest Browser Mode) and a Playwright adapter share the same core and are in progress.
