FAQ

Questions that come up

The short answers. For the longer ones, the concepts page covers the model and the API page covers the surface.

Why won't it run under jsdom?

Because jsdom has no layout engine, and the analysis needs one. tabbable uses visibility to decide what is in the tab order, and the visibility and visual-order rules read live bounding rects. Under jsdom those reads are meaningless, so a green result there would not tell you anything true. Run it in a real browser, headless or not.

Does it work with shadow DOM?

Yes, for open shadow roots. The analyzer asks tabbable to descend into them (getShadowRoot: true), so focusable elements inside an open shadow tree land in the sequence and get numbered. Closed roots are opaque to everyone, including the browser's own focus walk, so nothing can see into them. The what's-tabbable demo includes a live shadow-DOM stop.

Why does a control have no number on the overlay?

Because it isn't a tab stop. An element can be focusable by script or a mouse click without being tabbable. A tabindex="-1", a disabled state, an [inert] ancestor, or being hidden all drop it out of the order, so the overlay gives it no number. A few things are tab stops in the browser but left out on purpose, like an <iframe> or a focusable scroll container, and the tabbable demo walks every one of these cases. If a control should be a stop and isn't, that is often the bug, and a rule like clickable-not-focusable will flag it with a ⊘.

How do I silence a rule?

Switch it off in the per-rule overrides, the same place you would re-grade one. 

analyzeTabOrder(document.body, {
  rules: { "prefer-native-element": "off" },
});

// Or through the overlay.
reveal({ analyze: { rules: { "prefer-native-element": "off" } } });

Will warnings fail my build?

No. Only error-severity findings flip result.valid, and warnings are advisory. If you gate on valid, warnings ride along in violations without breaking the build. Promote a warning to an error (or demote an error) per call if your policy differs. See error vs warning.

Can I use it with React, Vue, or Svelte?

Yes. @focuspocus/reveal touches nothing but the DOM. Create it on mount, and call destroy() on unmount. The recipes page has React and Vue snippets. The analyzer itself is just a function over a DOM node, so it fits any stack.

Does the overlay interfere with my page?

It draws into its own layer and marks every node it owns, so the analyzer skips its own badges instead of mistaking them for page content. It reads your DOM but doesn't mutate it, and it redraws only when the verdict actually changes (geometry is tracked separately), so even a list that rewrites its rows every frame stays cheap.

Is there a test matcher or a Playwright adapter?

In progress. Both share the same core analyzer, which is why it is kept light on dependencies so it runs inside a headless browser or a Playwright page.evaluate. Until they land, assert on the result of analyzeTabOrder directly in a real-browser test runner. The recipes page shows how.
