Getting started

From clone to overlay in a minute

There are two ways in. You can run this site's interactive demos locally to see the overlay at work, or you can drop the analyzer into your own page. Both run in a real browser, which is the whole point. The test-runner matcher is a separate package that is still in progress, so it is not covered here yet.

Run the demos locally

The fastest way to understand the project is to run the two demo pages and tab through them with the overlay on. 

git clone https://github.com/focuspocusjs/focuspocus
cd focuspocus
pnpm install
pnpm --filter @focuspocus/demo dev

Open the printed URL. You do not need to build @focuspocus/core or @focuspocus/reveal first. The demo's vite.config.ts aliases both packages to their TypeScript source, so edits to the library hot-reload straight into the page.

Which package, and which mode

One analyzer, two ways to run it. See it on a live page with the overlay, or check it headless in a test, in CI, or from an AI agent. Installing @focuspocus/reveal covers both: it re-exports the analyzer, so the overlay and the headless API come from a single import.

See it, the overlay:

npm i @focuspocus/reveal

import { reveal } from "@focuspocus/reveal";
reveal();

Check it, headless:

npm i @focuspocus/core

import { audit, formatViolations } from "@focuspocus/core";
const result = audit(document.body);

Reach for @focuspocus/core on its own only when you never draw the overlay, for a test-runner matcher or a Playwright page.evaluate, where the smaller install matters.

Add the overlay to your page

@focuspocus/reveal is the shipped, framework-agnostic overlay. Import it and call reveal(), and you have numbered tab stops, the path between them, and every finding ringed in place. 

import { reveal } from "@focuspocus/reveal";

// No options, so it analyzes the whole document.
const overlay = reveal();

// Later, tear it down. This removes the layer, observers, and listeners.
overlay.destroy();

The overlay re-analyzes itself on every DOM mutation, so most pages need nothing beyond the mount. You can scope it to a subtree, hide and show it, or feed it rule overrides through reveal(options). See the reveal API.

The packages target npm as @focuspocus/core and @focuspocus/reveal. Until they are published, consume them from the workspace (or a git install) as the demo does.

Only need the verdict, not the picture? Skip the overlay and call audit() directly, covered just below.

Analyze without the overlay

The overlay is optional. @focuspocus/core gives you the verdict as plain data, which is what you would assert in a test or feed into your own UI.

Want to see these findings on the page instead of asserting them? Mount the overlay with reveal(), covered just above. 

import { audit, formatViolations } from "@focuspocus/core";

const result = audit(document.body);

result.valid; // boolean, true when there are no error-severity violations
result.sequence; // elements in tab order, each with tabIndex + rect
result.violations; // [{ rule, severity, message, element, selector, ... }]

if (!result.valid) {
  console.log(formatViolations(result.violations));
}

valid only goes false on an error. Warnings are advisory and never flip it, so they will not fail a build gate unless you promote them. See concepts for what that distinction means, and the core API for every field.

Turn rules up, down, or off

Every rule runs at a default severity. You can override any of them per call. Re-grade a noisy one to a warning, promote another to an error, or switch one off entirely. 

audit(document.body, {
  rules: {
    "visual-order-mismatch": "warning", // re-grade
    "redundant-tabindex": "error", // promote
    "prefer-native-element": "off", // disable
  },
});

// The same overrides reach the overlay through its `audit` option.
reveal({ audit: { rules: { "prefer-native-element": "off" } } });

The full list, with what each rule catches, is on the rules page.
