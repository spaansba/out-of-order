Recipes

Putting it to work

A handful of short, practical patterns. Gate a build on the verdict, scope the overlay, mount it in a framework, add a rule, and turn a finding into a fix. All of it runs through the same two entry points, audit and reveal.

Gate a build in CI

The analyzer needs a real DOM, so run it in a real-browser test runner like Vitest Browser Mode or Playwright. A dedicated assertion matcher is in progress. Until it lands, assert on the result directly, which works in any real browser today. 

// A test running in real Chromium (e.g. Vitest Browser Mode).
import { test, expect } from "vitest";
import { audit, formatViolations } from "@focuspocus/core";

test("modal has a sane tab order", () => {
  document.body.innerHTML = `
    <button>First</button>
    <a href="#">Second</a>
    <input aria-label="Search" />
  `;

  const result = audit(document.body);
  // The formatted violations become the failure message.
  expect(result.valid, formatViolations(result.violations)).toBe(true);
});

Because only error-severity findings flip valid, warnings will not fail the gate. Promote or demote rules per call if your policy differs (see changing a rule's grade).

Scope the overlay, or run two

Point reveal at a subtree with root, and use exclude when a second overlay owns part of the page, so a region isn't numbered twice. 

// Scope the overlay to just the app shell.
reveal({ root: document.querySelector("#app") });

// Two overlays without double-numbering. One is dedicated to a dialog, and the
// page overlay is told to exclude that same subtree.
const dialog = document.querySelector(".dialog");
reveal({ root: dialog });
reveal({ exclude: dialog });

Mount it in a framework

reveal is framework-agnostic. Create it on mount, and call destroy() on unmount. That is the whole lifecycle.

React

import { useEffect } from "react";
import { reveal } from "@focuspocus/reveal";

useEffect(() => {
  const overlay = reveal();
  return () => overlay.destroy();
}, []);

Vue

import { onMounted, onBeforeUnmount } from "vue";
import { reveal } from "@focuspocus/reveal";

let overlay;
onMounted(() => (overlay = reveal()));
onBeforeUnmount(() => overlay?.destroy());

Gate it behind a dev-only flag if you don't want the overlay in production builds.

Add a custom rule

A rule is a pure function over the sequence. Pass yours through the rules option and it runs with the built-ins, both in the result and on the overlay. 

import { reveal } from "@focuspocus/reveal";
import type { Rule } from "@focuspocus/core";

const noUnsafeBlank: Rule = {
  id: "no-unsafe-blank",
  docs: "https://html.spec.whatwg.org/multipage/links.html#link-type-noopener",
  defaultSeverity: "warning",
  run: (sequence) =>
    sequence
      .filter(
        (entry) =>
          entry.element.matches('a[target="_blank"]') &&
          !(entry.element.getAttribute("rel") ?? "").includes("noopener"),
      )
      .map((entry) => ({
        message: 'Link opens a new tab without rel="noopener".',
        target: entry,
      })),
};

reveal({ rules: [noUnsafeBlank] });

The full Rule, RuleRun, and Finding types are on the API page.

Turn a finding into a fix

Every violation is self-describing. It carries the rule it tripped, a message, a spec link, and the element (plus a selector for logs). Walk them and you have a punch list. 

const { violations } = audit(document.body);

for (const v of violations) {
  console.log(`[${v.severity}] ${v.rule} -> ${v.selector}`);
  console.log(`  ${v.message}`);
  if (v.docs) console.log(`  docs: ${v.docs}`);
}

When several controls share one root cause (a whole reimplemented widget, say), they arrive as one violation with the others in relatedElements, so your list has one entry to fix, not N. The violations demo pairs each finding with the exact before/after markup that clears it.
