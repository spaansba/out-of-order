Recipes

Putting it to work

A handful of short, practical patterns. Gate a build on the verdict, mount the overlay in a framework, add a rule, and hand the findings to an AI agent. All of it runs through the same two entry points, audit and trace.

Gate a build in CI

The analyzer needs a real DOM, so run it in a real-browser test runner like Vitest Browser Mode or Playwright. A dedicated assertion matcher is in progress. Until it lands, assert on the result directly, which works in any real browser today. 

// A test running in real Chromium (e.g. Vitest Browser Mode).
import { test, expect } from "vitest";
import { audit } from "@out-of-order/core";

test("modal has a sane tab order", () => {
  document.body.innerHTML = `
    <button>First</button>
    <a href="#">Second</a>
    <input aria-label="Search" />
  `;

  const result = audit();
  // A `text`-formatted audit's violations become the failure message.
  expect(result.valid, audit(document.body, { format: "text" }).violations).toBe(true);
});

Because only error-severity findings flip valid, warnings will not fail the gate. Promote or demote rules per call if your policy differs (see changing a rule's grade).

Mount it in a framework

trace is framework-agnostic. Create it on mount, and call destroy() on unmount. That is the whole lifecycle.

React

import { useEffect } from "react";
import { trace } from "@out-of-order/trace";

useEffect(() => {
  const overlay = trace();
  return () => overlay.destroy();
}, []);

Vue

import { onMounted, onBeforeUnmount } from "vue";
import { trace } from "@out-of-order/trace";

let overlay;
onMounted(() => (overlay = trace()));
onBeforeUnmount(() => overlay?.destroy());

Gate it behind a dev-only flag if you don't want the overlay in production builds.

Add a custom rule

A rule is a pure function over the sequence. Pass yours through the rules option and it runs with the built-ins, both in the result and on the overlay. 

import { trace } from "@out-of-order/trace";
import type { Rule } from "@out-of-order/core";

const noShouting: Rule = {
  id: "no-shouting",
  defaultSeverity: "warning",
  run: (sequence) =>
    sequence
      .filter((entry) => {
        const text = entry.element.textContent?.trim() ?? "";
        return text !== "" && text === text.toUpperCase();
      })
      .map((entry) => ({
        message: "This label is ALL CAPS, which reads as shouting. Use sentence case.",
        target: entry,
      })),
};

trace({ rules: [noShouting] });

The full Rule, RuleRun, and Finding types are on the API page.

Hand the findings to an AI agent

trace keeps its latest analysis on the handle as result, refreshed as the DOM changes. It's plain, serializable data, so any agent SDK can consume it. Format the findings into a readable brief and let a model draft the fixes. 

import { trace, audit } from "@out-of-order/trace";
import Anthropic from "@anthropic-ai/sdk";

const client = new Anthropic();
const overlay = trace();

async function reviewA11y() {
  const result = overlay.result;
  if (!result || result.valid) return;

  // `format` renders the result's violations to a ready-to-print string.
  const report = audit(document.body, { format: "text" }).violations;
  const message = await client.messages.create({
    model: "claude-opus-4-8",
    max_tokens: 1024,
    messages: [
      {
        role: "user",
        content: `You are an accessibility engineer. These are the tab-order ` +
          `findings on the current page. For each, propose the smallest ` +
          `markup change that clears it.\n\n${report}`,
      },
    ],
  });

  const [block] = message.content;
  if (block?.type === "text") console.log(block.text);
}

Each violation carries its element and selector plus the issues it failed (rule, message, spec link), so you can also serialize result as JSON for an MCP tool or any other agent framework. Keep the API key out of client bundles by proxying the call through your own server.
