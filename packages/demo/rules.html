Rules

Every rule, and the clause behind it

All rules are on by default. The severity shown is the default grade, and you can override any of them per call. The "layout" column marks the rules that read live CSS layout, the ones that can only run in a real browser. Every rule maps to a specific WCAG, WAI-ARIA, or HTML-spec clause, linked at the end of its row.

The rule set

Rule Default Layout What it catches Clause
no-positive-tabindex error no tabindex > 0, which hijacks the natural order WCAG 2.4.3
visual-order-mismatch warning yes tab order that doesn't follow top→bottom, left→right reading order WCAG 2.4.3
missing-accessible-name error no focusable interactive elements with no accessible name WCAG 4.1.2
aria-hidden-focusable error no tabbable element inside aria-hidden="true" WAI-ARIA
hidden-while-focusable error yes tabbable but invisible (opacity:0, zero size, off-screen, clipped) WCAG 2.4.7
clickable-not-focusable error yes mouse-only control (role/onclick) the keyboard can't reach WCAG 2.1.1
composite-roving-tabindex warning no composite widget (toolbar, tablist, ...) with one tab stop per item ARIA APG
focus-escapes-modal error no background controls still tabbable while a modal is open ARIA APG
tabindex-on-noninteractive error yes tabindex="0" on a role-less, non-interactive element ARIA APG
prefer-native-element warning no generic tag reimplementing a native control via an interactive role First rule of ARIA
duplicate-autofocus warning yes more than one focusable autofocus (only the first ever wins) HTML spec
autofocus-not-focusable warning yes autofocus on a non-focusable element (a no-op on load) HTML spec
nested-interactive error no a focusable control nested inside another focusable element WCAG 4.1.2
redundant-tabindex warning no tabindex="0" on an already-focusable native control (a no-op) HTML spec

The defaults are exported as DEFAULT_SEVERITY, a Record<RuleId, Severity>, so your own tooling can read the same source of truth the analyzer uses.

Changing a rule's grade

Pass per-rule overrides through AnalyzeOptions.rules. A rule you do not list keeps its default. For any other rule, re-grade it to "error" or "warning", or switch it "off". 

analyzeTabOrder(document.body, {
  rules: {
    "visual-order-mismatch": "warning",
    "redundant-tabindex": "error",
    "prefer-native-element": "off",
  },
});

Only error-severity findings flip result.valid, so demoting a rule to a warning keeps it reported without failing a gate. The same overrides reach the overlay through reveal({ analyze: { rules: ... } }).

Adding your own rule

A rule is just a pure function over the computed sequence. Write one and pass it alongside the built-ins, and it shows up in the result and on the overlay like any other. See the custom-rule recipe and the Rule type.
