Concepts

How a tab order becomes a verdict

Out of Order does two things in a row. First it works out the path the Tab key takes through a page. Then it grades that path against a set of rules. Once you see that split, and why both halves need a real browser, you know most of what there is to know.

The tab sequence

The first half is the order itself. It is every element the keyboard lands on, in the order it lands on them. Out of Order gets this from tabbable, which walks the DOM (and open shadow roots) and decides what is in the tab order right now. It accounts for more than tabindex. It also handles disabled state, inert subtrees, hidden ancestors, and the quirks of radio groups, <details>, and media elements.

A few things sit in the tab order in real browsers but are left out on purpose, because they cannot be reasoned about across browsers. One is an <iframe>, since its own document owns its stops. Another is a keyboard-focusable scroll container, since browsers disagree on it. The what's-tabbable demo walks every one of these edge cases, with the overlay numbering the real sequence.

From sequence to verdict

A sequence on its own is not right or wrong. The second half is the rules layer, which asks four questions about it. Every rule Focus Pocus ships answers exactly one.

  • Order is about whether focus moves in reading order (top to bottom, left to right), or whether a positive tabindex or a mismatch between the DOM and the paint scrambles it.
  • Reachability is about whether the keyboard can reach every interactive control, and whether it skips the dead stops, like decorative boxes or a no-op tabindex, that it should not catch.
  • Visibility is about whether every stop is actually painted, or is tabbable while invisible because of opacity:0, zero size, an off-screen position, or clipping.
  • Announcement is about whether every stop carries an accessible name, and whether it is clear of the aria-hidden, nesting, and modal traps that a screen reader stumbles on.

Each rule is grounded in a specific WCAG, WAI-ARIA, or HTML-spec clause. The rules page lists all of them with the clause and a live example.

Error vs warning

Every finding carries a severity. The line between the two is about impact, not certainty.

Severity Means Affects valid?
error A real barrier. A control is unreachable, unannounced, or invisible, or focus is trapped or leaking. Yes. One error makes valid false.
warning Dead or no-op markup, or a best-practice nit that doesn't block anyone (a redundant tabindex, a reimplemented native control). No. Advisory only, and never flips valid.

Because only errors flip valid, you can gate a build on it without warnings tripping the wire. Any rule's grade is overridable per call, so this split is a starting point, not a fixed policy. See turning rules up, down, or off.

Reading the overlay

The overlay is a direct picture of the result above, and nothing in it is decorative. Here is what each mark means.

  • A numbered badge is a tab stop, in the exact order the keyboard reaches it. Number 1 is the first stop.
  • A line connects each stop to the next. It turns red when that hop runs against reading order, so a zig-zag path is visible at a glance.
  • A ring around an element marks a finding. Its colour is the worst severity on that element, so red means an error and amber means a warning.
  • An off-sequence marker (a ⊘ instead of a number) flags a control that violates a rule but isn't a tab stop at all, like a clickable <div> the keyboard can never reach.
  • An element with autofocus is marked where load-focus lands, so you can see whether the page grabs focus and where.
  • Hover any marker to see the details. You get its accessible name, role, resolved tabindex, and every rule it trips.

Why it has to be a real browser

Both halves depend on layout. tabbable needs to know what is visible to decide what is in the order, and the visibility and visual-order rules read live bounding rects to know what is painted and where. jsdom has no layout engine, so it cannot answer either question. A control that is clipped to nothing, or that is visually first but last in the DOM, looks identical to a correct one.

So Out of Order runs every check against real layout. The cost is that it needs a real DOM, a browser or a headless one. The payoff is that a passing result actually means a keyboard user is fine. The analyzer stays light enough to run inside a headless browser or a Playwright page.evaluate for exactly this reason.
